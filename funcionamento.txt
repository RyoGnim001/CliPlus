#Fluxo da aplicação 

Frontend → POST /auth/login (Supabase direto, sem passar pelo ApiFlask)
                    ↓
             Supabase devolve JWT
                    ↓
Frontend → qualquer endpoint do ApiFlask enviando JWT no header
                    ↓
         ApiFlask valida o JWT com o Supabase
                    ↓
              Executa a lógica


#Organização

CLIPPLUS/
├── app/
│   ├── database/       → conexão com Supabase (supabase_client.py)
│   ├── helpers/        → funções utilitárias (ex: decorator requer_auth)
│   ├── models/         → modelos de dados (classes Python representando tabelas)
│   ├── repository/     → queries no banco (camada que fala com o Supabase)
│   ├── routes/         → endpoints do ApiFlask (auth, pacientes, consultas)
│   ├── schemas/        → validação de entrada/saída dos endpoints (Marshmallow/Pydantic)
│   └── services/       → lógica de negócio (ex: lógica de dedup de paciente)
├── settings/
├── .env                → variáveis de ambiente (SUPABASE_URL, SUPABASE_SERVICE_KEY)
└── app.py              → inicialização da aplicação

#Chamada

route → service → repository → database

#fluxo do campo CPF

Boa pergunta. Isso é uma decisão de frontend — o backend não faz isso sozinho. Deixa eu explicar como vai funcionar na prática:
O frontend vai ter dois momentos:
Momento 1 — quando o operador digita o CPF:
O frontend chama GET /pacientes/{cpf} automaticamente (ex: quando o campo CPF perde o foco). Dois cenários:

Retornou dados → preenche os campos do formulário automaticamente
Retornou 404 → não faz nada, deixa os campos em branco para o operador preencher

Momento 2 — quando o operador clica em salvar:
O frontend sempre chama POST /consultas com tudo que está preenchido no formulário — independente se o paciente já existia ou não. O backend resolve internamente:
POST /consultas recebe o CPF
        ↓
busca o CPF no banco
        ↓
Existe? → usa o id do paciente existente
Não existe? → cria o paciente com os dados do formulário
        ↓
registra a consulta
Ou seja, o frontend só faz duas chamadas simples: GET para checar o CPF e POST para salvar. Toda a lógica de "criar ou reaproveitar paciente" fica no backend, dentro do POST /consultas. O frontend não precisa saber se o paciente é novo ou não.
